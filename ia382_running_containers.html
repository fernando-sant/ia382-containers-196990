Good practices
  1. Name your containers. Give containers meaningful names with --name to identify them easily.
docker run --name my-alpine alpine

As before, the container stops immediately. List all containers to see it:

docker container ls -a
  1. Auto-remove one-off containers. Use --rm to avoid disk clutter.
docker run --rm --name container-to-remove alpine

Listing containers again should show that container-to-remove was not kept.

  1. Remove unused containers. Delete stopped containers explicitly, or prune them.
docker container rm my-alpine
docker container prune
  1. Remove unused images. Images take disk space; clean up when not needed.
docker image rm IMAGE_NAME
docker image prune -a

1.1 ¬∑ Pass a command to the containerized application

Template:

docker run [options] image-name [command] [arg]

The optional command is executed inside the container (guest terminal). For alpine, the default command is /bin/sh.

Notice

Host terminal: your computer‚Äôs terminal.
Guest terminal: the terminal inside the container.

2 ¬∑ Creating images

A Docker image is a stack of immutable layers. A container adds a writable layer on top. Files created at runtime (e.g., the page downloaded in Exercise 1.7) live in the container‚Äôs writable layer.

2.1 ¬∑ Dockerfiles

We‚Äôll create an image down-figures to download figures from a web page.

FROM ubuntu
RUN apt-get update
RUN apt-get install -y wget
RUN mkdir -p /my-figures
WORKDIR /my-figures
ENTRYPOINT ["wget", "-nd", "-r", "-A", "jpg,jpeg,bmp,png,gif"]
CMD ["https://feec-seminar-comp-eng.github.io/seminars/seminars-2-2025/"]
  • FROM: base image.
  • RUN: install dependencies and prepare directories.
  • WORKDIR: set working directory.
  • ENTRYPOINT: default executable and options.
  • CMD: default argument (URL) if none is given at runtime.

3 ¬∑ Data volumes

In Exercise 2.3, the pictures were downloaded into /my-figures inside the container. To transfer files to the host, use volumes.

3.1 ¬∑ Docker volumes

Create a volume:

docker volume create data-volume
Good to know (advanced)

Inspect the volume to see the mount point:

docker volume inspect data-volume

On Linux, the path exists on the host. On macOS/Windows (Docker Desktop), it lives inside the hidden VM.

To explore the VM:

docker run -it --rm --privileged --pid=host justincormack/nsenter1

docker network connect rochefort porthos
docker start -ai porthos
